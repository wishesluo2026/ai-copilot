01 · Project Background

项目背景：高规范写作场景的效率困境

体制内写作具有高度规范性、政策依赖和结构化要求，传统写作方式难以在效率与合规之间取得平衡。

Scenario 01

党建材料

党支部工作报告、年度总结、党员教育材料等，格式严格，政治表达有特定规范。从零起草平均耗时 4–8 小时，容错率极低。

Scenario 02

公文报告

请示、报告、通知、函等正式公文，行文结构、标题层级、落款规范均有明确要求，不熟悉规范者极易出错，每次提交都面临退稿风险。

Scenario 03

会议纪要

会议产生大量信息，手动整理耗时耗力，关键决议和行动项容易遗漏，纪要质量参差不齐，影响后续落实。

Why This Problem Is Worth Redesigning

为什么这是一个值得被重新设计的产品问题

Scale

全国机关事业单位工作人员超过 5000 万。写作是绝大多数岗位的高频必备技能，占工作日时间估计在 20–40% 之间。这不是小众场景，是被长期低估的大规模效率问题。

Pain

现有工具（Word 模板、通用 AI）对体制内写作规范的理解几乎为零——前者不智能，后者不专业。两类工具都无法同时满足"格式合规、语境准确、数据不出内网"三项基本要求。

Timing

大模型中文写作能力在 2023–2024 年出现明显提升，配合 RAG 与规则约束解码，具备了构建专业写作助手的技术基础。时间窗口相对明确，先进入的产品有积累优势。

04 · Solution Design

系统架构：三层设计，每层有明确的产品决策依据

核心判断：体制内写作的复杂性无法靠"更好的 Prompt"解决，必须在系统层面内置专业知识与规范约束。

Layer 01 · Interaction

用户交互层

混合结构化表单与对话界面，让用户以最低认知成本表达需求，同时保证系统获取足够精准的生成信号。

Input Design

场景引导式输入

  • 场景类型选择（党建材料 / 公文报告 / 会议纪要）
  • 结构化关键要素填写，降低用户表达负担
  • 参考文档 / 会议录音附件上传
  • 语气风格偏好设置
设计意图：用结构化输入替代 Prompt 技巧，让不会写 Prompt 的用户也能精准驱动生成。
Output Design

透明可审的输出

  • 流式生成，实时呈现写作过程
  • 格式规范问题内联标注，可见可修
  • 历史版本对比与一键回滚
  • Word / PDF 标准导出
设计意图：输出可见性降低用户对"AI 黑盒"的不信任感，建立可审核的人机协作流程。
Layer 02 · Understanding

场景理解层

系统自动完成意图解析与格式匹配。用户不需要了解 RAG 是什么，也能获得专业精准的支持。

Intent Engine

意图识别与要素提取

  • 场景分类模型（内测准确率 >95%）
  • 关键要素自动结构化提取
  • 多轮对话上下文连续追踪
  • 格式模板智能匹配
设计意图：把"理解用户真实需求"的负担从用户侧转移到系统侧。
RAG Retrieval

知识库检索增强

  • 机构历史文档私有检索库
  • 公文规范知识图谱
  • 政策术语与惯用表达词典
  • 相似案例语义向量检索
设计意图：没有 RAG，通用大模型无法"懂规矩"——生成质量将无法满足合规要求。RAG 是专业化的核心。
Layer 03 · Generation & Control

生成与质控层

专域微调 + 规则约束解码 + 闭环反馈。确保输出文风符合体制内规范，并通过用户行为持续迭代。

Generation Model

专域微调大模型

  • 10 万+ 体制内写作语料监督微调（SFT）
  • 机关文风偏好人类反馈对齐（RLHF）
  • 格式规范约束解码策略
  • 敏感内容过滤与合规性检查
设计意图：通用模型在这个场景必须经过领域微调——Prompt 无法替代领域知识的内置。
Quality Control

输出质控与反馈闭环

  • 200+ 项格式规范自动校验
  • 前后文语义一致性检查
  • 用户修改行为采集与分析
  • 异常样本人工标注进入训练迭代
设计意图：规则约束确保下限，反馈闭环持续抬升上限——产品越用越准。

05 · Capability Design

核心能力矩阵

按用户价值层级划分：Tier 1 直接解决核心问题，Tier 2 降低使用门槛，Tier 3 构建长期护城河。

Tier 1 · Core Differentiation  核心差异化能力，无法被通用工具替代
解决 P1
✍️

场景化文档生成

专域模型深度理解公文场景，输入结构化要素后秒级生成符合规范的完整初稿，支持 20+ 种公文文体，显著压缩起草时间。

解决 P2
🔍

格式规范自动校验

内置 200+ 项公文格式规则引擎，内联标注格式问题，可一键修复。将审稿往返从平均 3 轮压缩至 1 轮以内。

高频场景
🎙

会议纪要全流程自动化

录音上传 → 转写 → 发言人识别 → 决议提炼 → 行动清单 → 规范纪要，端到端自动完成，无需人工逐字整理。

Tier 2 · Experience Enhancement  体验增强，降低使用门槛
体验增强
💬

对话式迭代修改

无需重新填写表单，通过自然语言指令（"第三段语气更正式"、"补充政策依据"）直接驱动局部修改，保持上下文连贯。

为什么这个能力属于 Tier 2 而非 Tier 1？ 对话修改是体验差异，而非场景差异——用户在没有它的情况下仍能完成任务。它的价值是降低摩擦，而不是解决根本问题。这个判断直接决定了它的研发优先级：低于格式校验与文档生成，高于知识库基础设施。
Tier 3 · Infrastructure  基础设施能力，随时间增值、构建护城河

08 · Reflection

Key Takeaways

三个在这个项目中形成的判断，它们不只适用于体制内写作，也适用于更广泛的 AI 垂直产品设计。

01

垂直场景的护城河是"内置的专业性"，不是功能数量

通用 AI 工具在体制内写作场景的局限，根本原因不是 Prompt 不够好，而是格式规范、政治术语、机构风格这三类知识根本没有内置。判断"是否值得做垂直产品"的核心问题，不是"通用工具能否满足需求"，而是"需求的规范密度是否高到通用工具无法用 Prompt 补偿"。体制内写作恰好跨越了这条线，这使得垂直化设计的投入是值得的。

Validated By
72:1
时间压缩比（6h → 5min），专业化设计密度的直接量化
02

AI 产品的人机协作边界，由用户心理决定，而非技术能力

早期设计追求 AI 直接输出可用稿，但调研发现用户对"AI 全权生成的内容"存在明确的信任障碍——在政务场景中尤其体现为合规性顾虑，而非能力质疑。将产品定位从"AI 替代写作"调整为"AI 生成高质量初稿，人工做最终判断"后，用户接受度从 41% 提升至 87%。设计 AI 产品时，最关键的判断不是"AI 能做什么"，而是"用户愿意让 AI 做什么、做到哪一步"。

Acceptance Rate
87%
重新定位后 vs 原定位 41%，差距源于对用户心理边界的准确判断
03

知识库是将线性工具变为复利平台的机制，需要在设计期就埋入

大多数 SaaS 工具的价值是线性的：你用它，它帮你；不用，它不变化。知识库改变了这个结构——每次使用都在沉淀数据，产品对机构的理解持续增长，用户替换成本持续上升。这种"越用越难替换"的特性，才是 B2G SaaS 的真实护城河。知识沉淀路径必须在产品设计阶段规划好，不能当作后期运营工作处理。

Retention Signal
68%
知识库活跃用户 7 日留存 81%，非活跃 52%，差距证明护城河机制已在起效

AI 产品经理的核心判断力不在于会用 AI，而在于能回答三个问题：哪些场景的专业密度足够高，值得专门设计；在这个场景中，AI 的介入边界应该划在哪里；如何让系统越用越好，而不是越用越让人依赖。

体制内写作恰好是三个问题都有清晰答案的场景——专业密度极高，边界在初稿而非终稿，知识可积累。正是因为这个场景的规范壁垒，使得真正做专的产品有机会创造通用工具无法替代的价值。
